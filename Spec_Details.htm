At Level 5 Form instances are stored for each document TYPE of INVOICE, CREDIT NOTE, PROOF of DELIVERY and PURCHASE ORDER

For SALES
A Purchase (Sales) Order received from the customer should start the process, however this is not always the case.
All subsequent "child" documents (Invoices, POD's, Credit Notes) should reference this Purchase Order. This referencing is via a FIELD on the child document.
(i.e. all of these forms have a CUSTOMER PURCHASE ORDER field on them   note: this field may be blank).

Credit Notes and POD's could be considered children of a specific Invoice. This referencing is via a FIELD on the child document.
(i.e. all of these forms have an OUR INVOICE NUMBER field on them   note: this field may be blank).

BPA.net form instances (and document images, if scanned) as stored under the level 4 (month) node by DATE OF ISSUE (a field on the Form).
This applies to ALL form/document TYPES.

For PURCHASES
Same issues as for sales (documents go in the opposite direction).

Retrieving & Viewing a DOCUMENT SET
One of the key benefits of the portal is that it allows all related Document TYPES to be associated together AUTOMATICALLY.
The user can select any document from the grid (on GRID1.aspx) and select the DOC SET button to view the set of all related Documents.
The set of documents is therefore presented in 4 grids on the DOCSET.aspx page (1 grid for each document TYPE).

Status and Synchronise control
"Documents" can be input directly into the system from within the portal (using Forms Supplied). The system must be configured with this privledge.
If document are put in this way then the data must be Synchronised to the Accounts package.

Documents may be "Published" to the Cash Portal from the VT-Scan program. Both Images and Form data is published to the system.
If documents are entered this way then the data must also be synchronised to the Accounts package.

Documents may be entered onto the Customers accounts system and Synchronised from there to the Cash Portal.

Documents (Forms) may be edited on the cash portal (if system is configured to allow this).
If the document is edited then the data must be re-synchronised with the accounts package.

THEREFORE there are PAYMENT status associated with forms and SYNCHRONISE status associated with forms.

PAYMENT STATUS (ideally these status could be defined by the customer)
 CURRENT
OVERDUE
PAID

DISPUTED

 SYNCHRONISE STATUS
WAITING SYNC.
UPLOAD TO ACCOUNTS
DOWNLOADED FROM ACCOUNTS

 There is also a document TYPE and SOURCE.
SOURCE will indicate where the document originated.
TYPE will indicate what TYPE the document is (e.g. a Scanned image, Web form)

Synchronising will be via a web SERVICE function call.
The service will read and write records (for each portal) to a local transaction log database.


System Adminstration facilities
The system adminstrator can do the following (to a specific customer portal)
  1. Add a New Customer as a user to the system
  2. Assign and manage Customer Passwords and Viewing Privledge
  3. Define document STATUS Types
  4. Set some workflow parameters (e.g. Frequency of reminder emails, etc.)
  5. Add New Employees As users to the system
  6. Assign and Manage Employee Passwords and Viewing Privilege
  7. Control Synchronisation issues (with accounts package etc.)
  8. Log a Support Request to Verify Technologies
  9. Review system status and audit logs
  10. Archive Older records
  11. Maintain SUPPLIER, CUSTOMER and EMPLOYEE database (modules)
  12. Review reports and system statistics
